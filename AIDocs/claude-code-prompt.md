# Claude Code Prompt: InformedVoter — US Civic Information Platform

## PROJECT OVERVIEW

Build a full-stack voter information web application called **InformedVoter** that empowers US citizens to make informed voting decisions. The app auto-detects a user's state via IP geolocation and displays relevant federal + state election data — candidates, policies, bills, voting records — organized by category. Users can manually override their detected state. The site must feel trustworthy, nonpartisan, and easy to navigate without overwhelming the user.

**Phase 1 (this build):** Federal + State level data
**Phase 2 (future):** Local elections (mayors, city council, school board, etc.)

---

## TECH STACK

### Frontend
- **Next.js 14+** (App Router) with TypeScript
- **Tailwind CSS** for styling
- **Framer Motion** for subtle, professional animations
- **Lucide React** for icons
- **React Query (TanStack Query)** for data fetching/caching

### Backend
- **Next.js API Routes** (serverless functions)
- **PostgreSQL** via **Supabase** (free tier → scales affordably)
- **Prisma ORM** for database access
- **NextAuth.js** for optional authentication (Google, email magic link)

### AI Layer
- **Anthropic Claude API**:
  - `claude-haiku-4-5-20251001` for bulk tasks (bill summaries, candidate policy extraction)
  - `claude-sonnet-4-20250514` for deep analysis (hidden clauses/riders, pros-cons generation)
- AI outputs are **cached in the database** to minimize API costs
- All AI-generated content displays a clear disclaimer: "This analysis was generated by AI and reviewed for accuracy. Always read primary sources."

### Data Sources (Free Tier APIs)
- **Congress.gov API v3** (bills, CRS summaries, members, actions) — free, official, 5000 req/hr
- **@unitedstates/congress-legislators** (GitHub) — photos, social media, contact info for all members (public domain)
- **Open States / Plural Open API v3** (state legislators, state bills) — free API key
- **LegiScan API** (state + federal bills, full bill text, roll calls) — free tier, 30K queries/mo
- **Google Civic Information API** (elections, polling places, ballot info — NOTE: Representatives endpoint was shut down April 2025, only elections/voterinfo/divisions remain)
- **OpenFEC API** (campaign finance data) — free, official
- **VoteView (UCLA)** (complete Congressional voting records since 1789) — free CSV downloads
- **Ballotpedia** (scraped for nonpartisan candidate/governor/ballot measure info — respect robots.txt, cache aggressively)
- ~~**ProPublica Congress API**~~ — SHUT DOWN, do not use
- **Vote.org** (embeddable voter registration/absentee ballot widgets) — free

### Deployment
- **Vercel** (hosting + serverless) — free tier to start, ~$20/mo at scale
- **Supabase** (PostgreSQL + auth) — free tier to start
- **Upstash Redis** (caching layer) — free tier
- **Estimated monthly cost:** $50-150/mo at moderate traffic

## PAGE DESIGNS & FEATURES

### 1. Landing Page (`/`)

**On load:**
- Call IP geolocation API (ip-api.com free tier or ipinfo.io) to detect user's state
- Show a clean hero with: "Welcome, it looks like you're in **[State]**"
- Prominent "Change State" dropdown if detection is wrong
- Below: quick cards showing what's available (Upcoming Elections, Active Bills, Your Representatives)
- A "What every voter should know" section with links to:
  - Voter registration status checker
  - Upcoming election dates
  - Your polling place
  - Voter ID requirements for your state
  - How to request an absentee ballot

**Design direction:** Clean, authoritative, nonpartisan. Think: government site meets modern editorial design. Navy blue, white, subtle red accents. Typography should feel trustworthy — something like a serif for headlines (e.g., Libre Baskerville, Playfair Display) paired with a clean sans-serif body (e.g., Source Sans Pro, DM Sans). Avoid flashy colors or anything that looks partisan.

### 2. State Dashboard (`/state/[stateAbbr]`)

**Layout:** Section cards in a clean grid. NOT all info dumped on one page. Each card is a gateway:

| Section Card | Description |
|---|---|
| 🏛️ US Senators | 2 senators, party, quick stance summary |
| 🏛️ US Representatives | List by district, party breakdown |
| 🏛️ Governor | Current governor, party, key policies |
| 📜 Federal Bills | Active federal bills affecting this state |
| 📜 State Bills | Active state legislation |

| 📋 How to Vote | Registration, deadlines, ID requirements |

Each card shows a brief preview (2-3 lines) and a "View Details →" link.

### 6. Voter Info & Action Center (`/state/[stateAbbr]/voter-info`)

This page is the most actionable page on the entire site — it's where voters DO things,
not just read things. Integrates TurboVote (via embed/partnership) alongside your own
state-specific data.

**INTEGRATION STRATEGY: TurboVote + Your Own Data**

TurboVote (by Democracy Works) provides embeddable tools that handle the complex,
state-by-state variations in voter registration, absentee ballots, and election reminders.
Rather than building all of this from scratch (which would require maintaining 50+ state
rulesets), embed TurboVote and supplement with your own curated data.

**To become a TurboVote partner (free for civic orgs):**
- Contact: partners.turbovote.org
- You'll get a custom subdomain: `informedvoter.turbovote.org`
- You'll get embed codes for iframes that render TurboVote tools on your site
- TurboVote embeds resize dynamically and work on mobile

**Page Layout — "Voter Action Center":**

**Section A: "Take Action" Cards (top of page, prominent)**

Four large action cards in a 2x2 grid (mobile: stacked):

### 8. Polling Place Finder (`/polling-places`)

A dedicated, prominent page accessible from the main nav and every state page.

**Input Methods (user chooses one):**

1. **Use My Location** button — requests browser geolocation permission, reverse geocodes
   to get address, then queries for polling places
2. **Enter Address** — full street address input with autocomplete (Google Places API 
   or free alternative like Nominatim/OpenStreetMap)
3. **Enter Zip Code** — simpler option, uses zip centroid for approximate results

**How It Works (Technical):**

```
User enters address or shares location
        │
        ▼
┌─── Google Civic Info API ───┐
│ voterInfoQuery endpoint     │
│ - pollingLocations[]        │
│ - earlyVoteSites[]          │
│ - dropOffLocations[]        │
│ Each has lat/lon + address  │
└────────────┬────────────────┘
             │
             ▼
   Calculate distance from          ◄── Use Haversine formula
   user's location to each              or Google Distance Matrix
   polling location                      API for drive times
             │
             ▼
   Sort by distance (nearest first)
             │
             ▼
   Display as list + map
```

**Results Display:**

- **Map View** (default): Interactive map (Mapbox GL JS free tier or Leaflet + 
  OpenStreetMap) showing:
  - User's location (blue pin)
  - Polling places (red pins — election day)
  - Early vote sites (orange pins)
  - Drop-off locations (green pins)
  - Click any pin for details card

- **List View** (toggle): Each polling place as a card showing:
  - Location name (e.g., "Westview Elementary School")
  - Full address
  - Distance from you (miles, calculated client-side via Haversine)
  - Estimated drive time (if using Google Distance Matrix, otherwise just distance)
  - Polling hours
  - Type badge: "Election Day" | "Early Voting" | "Drop-off Only"
  - Start/end dates (for early voting sites)
  - "Get Directions" button (opens Google Maps/Apple Maps with directions)
  - Any special notes (accessibility, parking, etc.)

- Results are sorted **nearest to furthest** by default
- User can re-sort by: distance, name, type, hours

**Fallback Strategy (CRITICAL):**

Google Civic API polling data is **not always available** — it depends on when states 
publish their data (sometimes only 5-25 days before an election). Handle gracefully:

1. **If data IS available:** Show full polling place results as described above
2. **If data is NOT available (between elections or state hasn't published):**

     for the upcoming election. Check back closer to Election Day."
   - Provide direct links to the state's official polling place locator
   - Maintain a `statePollingLocatorUrls` table in your DB with each state's 
     official lookup tool URL (e.g., Florida: https://registration.elections.myflorida.com/CheckVoterStatus)
   - Show the "How to Vote" info for their state as a helpful fallback
   - Offer to notify them when polling data becomes available (if they have an account)

## DATA PIPELINE ARCHITECTURE

### Ingestion Service (Cron Jobs / Vercel Cron)

Create a `/api/cron/` directory with these scheduled jobs:

1. **`sync-members`** (daily) — Pull current members of Congress from Congress.gov API + OpenStates API for state legislators. Upsert into Candidate table.

2. **`sync-bills`** (every 6 hours) — Pull active bills from Congress.gov + LegiScan. Upsert into Bill table. Queue new bills for AI analysis.

3. **`sync-votes`** (every 6 hours) — Pull roll call votes from Congress.gov bill actions + LegiScan getRollCall + VoteView bulk data. Insert into BillVote table.

4. **`analyze-bills`** (runs after sync-bills) — For each new/updated bill without AI analysis:
   - Fetch full bill text
   - Send to Claude Haiku for executive summary
   - Send to Claude Sonnet for rider/hidden clause detection
   - Cache results in Bill table (executiveSummary, aiRiderAnalysis, hiddenClauses)
   - Rate limit: process 10 bills per run to stay within API budget

5. **`analyze-candidates`** (weekly) — For each candidate:
   - Aggregate their voting record, public statements, official platform
   - Generate policy position summaries using Claude
   - Generate "supporters say / critics say" perspectives
   - Cache in CandidatePolicy table

6. **`sync-voter-info`** (weekly) — Update VoterInfo table with registration deadlines, voting rules per state. Source: Vote.org API + manual curation for accuracy.

7. **`sync-elections`** (daily during election season, weekly otherwise) — Pull upcoming elections from Google Civic API.

8. **`sync-campaign-finance`** (daily) — For each federal candidate in the database:
   - Fetch financial totals: `/candidates/{fecCandidateId}/totals/`
   - Fetch top donors by employer: `/schedules/schedule_a/by_employer/?committee_id={id}`
   - Fetch contributions by size: `/schedules/schedule_a/by_size/by_candidate/?candidate_id={id}`
   - Fetch contributions by state: `/schedules/schedule_a/by_state/by_candidate/?candidate_id={id}`
   - Fetch top individual donors: `/schedules/schedule_a/?committee_id={id}&sort=-contribution_receipt_amount&per_page=20`
   - Fetch top expenditures: `/schedules/schedule_b/?committee_id={id}&sort=-disbursement_amount&per_page=20`
   - Fetch independent expenditures for/against: `/schedules/schedule_e/?candidate_id={id}`
   - Rate limit carefully: OpenFEC allows 1,000 req/hr with registered key
   - Strategy: process 50 candidates per run, rotate through all candidates over multiple days
   - Upsert into CandidateFinance, CandidateTopDonor, CandidateTopIndustry, 
     CandidateContributionBySize, CandidateContributionByState, CandidateExpenditure,
     and IndependentExpenditure tables

### Caching Strategy
- **Redis (Upstash):** Cache API responses for 1 hour. Cache AI analysis results for 7 days.
- **Database:** All AI-generated content stored in PostgreSQL so it's not re-generated.
- **ISR (Incremental Static Regeneration):** State dashboard pages regenerate every 30 minutes. Bill detail pages regenerate every 1 hour. Candidate pages regenerate every 24 hours.

```
You are a nonpartisan legislative analyst. Summarize this bill in 3-4 sentences that a 
high school student could understand. Include: what it does, who it affects, and its 
current status. Do not use partisan framing. Be factual and concise.

Bill Title: {title}
Bill Text: {text}
```

### Hidden Clause / Rider Detection Prompt
```
You are a nonpartisan legislative analyst specializing in identifying unrelated provisions 
(riders) in legislation. Analyze this bill and identify any sections that appear unrelated 
to the bill's primary stated purpose.

For each unrelated provision found, provide:
1. The section/subsection reference
2. A plain-English explanation of what it does
3. Why it appears unrelated to the bill's title and primary purpose
4. A severity rating: MINOR (tangentially related), SIGNIFICANT (loosely related at best), 
   or MAJOR (completely unrelated to the bill's purpose)

If no unrelated provisions are found, say so explicitly.

Bill Title: {title}
Bill Primary Purpose: {summary}
Full Bill Text: {text}

Respond in JSON format:
{
  "hasRiders": boolean,
  "riders": [
    {
      "section": "string",
      "explanation": "string",
      "whyUnrelated": "string", 
      "severity": "MINOR | SIGNIFICANT | MAJOR"
    }
  ],
  "notes": "string (any additional context)"
}
```

### Candidate Policy Analysis Prompt
```
You are a nonpartisan political analyst. Based on the following information about this

Structure your response as JSON:

  "supportersSay": ["point 1", "point 2", "point 3"],
  "criticsSay": ["point 1", "point 2", "point 3"],
  "aiAnalysis": "A balanced 1-paragraph analysis noting nuances, trade-offs, and context. 
    Do not advocate for or against the position.",
  "sources": ["source descriptions for fact-checking"]
}

Candidate: {name} ({party})
Office: {office}
Voting Record on {policyCategory}: {votes}
Official Platform Statements: {statements}
Public Statements: {publicStatements}
```

## KEY UI/UX PRINCIPLES

1. **Progressive disclosure** — Never show everything at once. Cards → sections → details.
2. **Nonpartisan design** — No red or blue dominance. Use navy/slate with equal party color accents.
3. **Mobile-first** — Most voters will use this on their phones.
4. **Speed** — ISR + Redis caching. Target <2s page loads.
5. **Accessibility** — WCAG 2.1 AA compliance. Screen reader friendly. High contrast mode.
6. **Trust signals** — Show data sources, last-updated timestamps, AI disclaimers everywhere.
7. **Shareable** — OG meta tags for social sharing. "Share this bill" buttons.
